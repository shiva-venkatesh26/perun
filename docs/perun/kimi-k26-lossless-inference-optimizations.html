Perun Inference Optimization Report

Research-backed optimization ideas for Perun, an AMD-first inference engine forked from SGLang. The starting model is assumed to be Kimi K2.6. The goal is higher throughput and GPU utilization without changing model quality.

Project: Perun Substrate: SGLang fork Starting model: Kimi K2.6 Target GPUs: AMD Instinct MI300X / MI325X / MI350X / MI355X Report date: 2026-06-01

Executive Verdict

Perun should optimize the parts of serving that are exact: scheduling, batching, KV reuse, verified speculation, communication, memory layout, and numerically equivalent kernels. Avoid claiming quality preservation for pruning, post-training quantization, approximate attention, KV dropping, or early exit unless the full target model verifies the result.

The first serious Perun wedge should be Kimi K2.6 on AMD with a better MoE path, better long-prompt scheduling, and a benchmark harness that proves quality did not move.

What "lossless" means here

Lossless means the optimization preserves the target model's greedy output or sampling distribution. A faster kernel can still change tiny floating-point details, so Perun needs logit-diff and end-to-end generation tests before calling it production-safe.

Practical rule: scheduling changes are usually safe; kernel changes need numeric tests; compression or approximation is not safe by default.

Model Constraints That Drive the Plan

Kimi K2.6 is documented by SGLang as a very large sparse MoE model with MLA attention, native multimodality, 64 attention heads, and a long context window. The AMD deployment note matters immediately: TP=4 is the safe path because AITER's MLA kernel requires heads_per_gpu % 16 == 0. With 64 heads, TP=4 gives 16 heads per GPU; TP=8 gives 8 and breaks that constraint. SGLang also notes that AMD FP8 KV cache is a memory/accuracy tradeoff, so Perun should not treat it as lossless by default.

Source: SGLang Kimi K2.6 deployment guide.

Top Optimization Backlog

1AMD Fused MoE Path

MoE routing and expert MLP execution will likely dominate Kimi decode. AMD's Kimi K2.5 profiling found fused_moe responsible for about 88-90% of GPU time before optimizing it with FlyDSL.

Highest leverage Kernel work Lossless if numerically equivalent

2Verified Speculative Decode

A small proposer drafts tokens; the full Kimi model verifies them. Done correctly, this reduces target forward passes while preserving the target distribution.

3Prefix and KV Reuse

Agent workloads repeat system prompts, tool schemas, repo summaries, and retrieved chunks. Exact KV reuse saves prefill compute without changing generation.

Prevents slow fallback kernels and avoids invalid AMD layouts. Numerically equivalent kernel path, but test it.
Verified speculative decoding SpecInfer, EAGLE, EAGLE-2/3, LongSpec, and heterogeneous-vocabulary speculative decoding all point to target-verified acceleration. Adaptive chunked prefill SARATHI and Sarathi-Serve show that chunking long prefills avoids decode stalls under latency targets. Tune and then automate chunked_prefill_size in python/sglang/srt/managers/scheduler.py based on prompt length, waiting decodes, and HBM pressure. Keeps decode work flowing instead of making GPUs wait behind giant prompts. Exact scheduling change.
KV-aware online scheduler Microsoft's May 2025 scheduling work models LLM inference under KV cache constraints and proposes online algorithms. Add a scheduler policy that accounts for KV footprint, request age, expected output length, and eviction pressure. Use schedule_policy and cache stats as the integration points. Better admission and batching means fewer OOM/preemption events and higher sustained utilization. Exact if request semantics are preserved.
Prefix cache admission and eviction Marconi shows cache policy matters for TTFT and token hit rate; SGLang already has RadixAttention/prefix-cache structure. Score cache entries by saved prefill FLOPs per HBM byte. Include model revision, tokenizer, chat template, and exact token IDs in cache keys. Repeated agent prompts become cheaper; fewer prefill FLOPs per customer request. Exact if cache identity is exact.
Disaggregated prefill/decode DistServe, Mooncake, and FlowKV separate compute-heavy prefill from memory/latency-sensitive decode. Use existing SGLang disaggregation modules first: python/sglang/srt/disaggregation. Add AMD-specific KV transfer metrics and admission control before deeper rewrites. Lets different GPU pools run at their natural utilization point, improving cluster goodput. Exact if KV transfer is exact.
MoE expert placement and communication MegaScale-Infer, DeepEP, and HarMoEny focus on expert-parallel communication, load imbalance, and GPU idle time. Add routing telemetry, expert hotness histograms, and an EP layout planner. Explore ROCm/RCCL equivalents for dispatch/combine and all-to-all paths. MoE economics depend on keeping all GPUs busy despite skewed expert routing. Exact if routing results and expert math are unchanged.
Operation-level overlap NanoFlow shows throughput gains from splitting work into smaller operation-level units and overlapping compute, memory, and communication. Start with measurement: add timelines for attention, MoE, collectives, sampling, and KV writes. Then prototype overlap in decode and MoE dispatch paths. Improves utilization inside a single GPU, not only across GPUs. Exact if execution order dependencies are respected.
Attention kernel benchmarking loop FlashInfer and PackInfer show that attention performance depends heavily on sequence length distribution and batching shape. Add a Perun benchmark suite for Kimi shapes: short prompts, 32K, 128K, 256K, mixed batch, decode-only, and speculative verifier attention. Prevents optimizing for the wrong shape; each percent matters at scale. Benchmarking is safe; kernel swaps need tests.

Research Notes, In Plain English

AMD AITER and FlyDSL: make AMD kernels first-class

The AMD story is not just "run PyTorch on ROCm." AITER provides AMD-specific fused kernels, and AMD's Kimi K2.5 work shows that MoE can dominate runtime. Perun should treat kernels as a product surface: select the right kernel per shape, measure it, and fail back safely.

Sources: ROCm vLLM optimization guide, AMD Kimi K2.5 FlyDSL blog.

Speculative decoding: speed up without changing answers

The trick is not to trust the small model. The small model only proposes. The full model verifies. If the verification math is correct, the final output follows the same target model distribution. For Kimi, begin with cheap prompt-lookup and n-gram proposers because code and agent traces repeat a lot.

Sources: SpecInfer, EAGLE, EAGLE-2, EAGLE-3, LongSpec.

Scheduling: feed the GPU the right mix

Inference is not just one big matrix multiply. Long prompts, short chats, and decode-heavy requests need different GPU resources. Chunked prefill, KV-aware scheduling, and online admission control aim to keep the GPU busy while avoiding tail-latency spikes.

Sources: SARATHI, Sarathi-Serve, Microsoft Research KV-cache scheduling.

KV cache: long context makes memory policy a core feature

KV cache is the model's memory of previous tokens. For 256K context, bad cache policy wastes HBM, causes preemption, and destroys TTFT. Exact prefix reuse is one of the cleanest lossless wins for agent workloads.

Sources: vLLM / PagedAttention, Marconi, Mooncake, FlowKV.

MoE systems: expert load imbalance is money leaking

Sparse MoE models save compute by activating a few experts, but serving gets harder because some experts become hot while others sit idle. Perun needs expert-routing telemetry before trying clever EP layouts.

Sources: MegaScale-Infer, DeepEP, Microsoft HarMoEny.

Intra-GPU overlap: look below batching

NanoFlow's core lesson is that a GPU can be underutilized even when the batch looks large. Memory, compute, and communication resources can be overlapped if the engine exposes smaller scheduling units. For Perun, this is a later-stage optimization after profiling tells us where bubbles are.

Source: NanoFlow.

What Not To Count As Lossless Yet

Idea Why It Is Risky How To Treat It
Extra post-training quantization Usually changes logits and output quality. Kimi's native INT4 checkpoint can be a baseline, but additional FP8/FP4/W4A8 conversions are not automatically lossless. Run it as a separate quality/performance track with benchmark gates.
KV compression, pruning, eviction, or sparsification Dropping or approximating KV changes attention inputs unless the model was trained for that behavior. Only use exact KV transfer/reuse in the lossless track.
Early exit or layer skipping Skipping target layers changes the model. Use only as a proposer inside verified speculative decoding.
Turning off thinking mode It changes the requested behavior and output distribution. Expose as a product latency option, not an engine optimization.
Approximate sparse attention Can be fast, but it changes attention unless the architecture defines that sparsity. Keep out of the first Perun lossless roadmap.

First 30 Days of Work

  1. Build a Kimi K2.6 profiling baseline: TP=4 on AMD, record per-kernel time for MoE, MLA, RoPE, RMSNorm, collectives, KV writes, sampling, and scheduler overhead.
  2. Add a correctness harness: compare logits, greedy outputs, seeded stochastic outputs, tool-call JSON, and reasoning parser behavior against upstream SGLang.
  3. MoE first: benchmark current Kimi MoE kernels and prototype a Kimi-shape registry for AITER/FlyDSL/Triton/CK candidates.
  4. Prefix cache telemetry: expose cache hit rate, saved prefill tokens, HBM bytes held, eviction churn, and TTFT impact.
  5. Speculation starter: implement prompt-lookup or n-gram speculative decode before training a custom drafter.
  6. Chunk-size sweeps: run 8K, 32K, 128K, and 256K prompt mixes to find AMD-specific chunked prefill settings.
  7. Benchmark vLLM ROCm as a control: Perun should beat strong baselines, not just its own previous version.

Metrics That Matter

Metric Why It Matters Report By
Tokens/sec/GPU Direct serving cost metric. Prefill, decode, mixed workload, speculative accepted tokens.
GPU utilization and idle bubbles Shows whether kernels, scheduler, or communication are starving the GPU. MoE, attention, collectives, CPU scheduling, KV transfer.
