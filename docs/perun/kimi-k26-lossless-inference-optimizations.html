Lossless Inference Optimization Map for Kimi K2.6 on AMD

This note collects output-preserving techniques Perun can productionize for Kimi K2.6. The emphasis is on optimizations that should preserve the target model distribution or exact greedy output, not techniques that trade quality for speed.

Project: Perun Model: Kimi K2.6 Hardware: AMD Instinct MI300X / MI325X / MI350X / MI355X Date: 2026-06-01

Executive Read

1Fused MoE on ROCm

Kimi is sparse MoE, so expert routing and expert MLP execution are likely the largest decode bottleneck. AMD's Kimi K2.5 work found fused_moe taking 88-90% of GPU time before replacing it with a FlyDSL mixed-precision fused MoE kernel.

Kimi K2.6 Constraints That Matter

Kimi K2.6 is a 1T-parameter sparse MoE model with 32B active parameters, MLA attention, 64 attention heads, native multimodality, and a 256K context window. Current SGLang guidance says AMD deployments use TP=4 because AITER's MLA kernel requires heads_per_gpu % 16 == 0; with 64 heads, TP=4 gives 16 heads per GPU. The same guidance recommends AMD ROCm images for MI300X/MI325X and MI350X/MI355X and notes that FP8 KV cache is a memory tradeoff rather than a strict lossless default. Source: SGLang Kimi K2.6 guide.

For this document, “lossless” means preserving the target model's sampling distribution or exact greedy output. Kernel substitutions can still introduce tiny floating-point differences; they need golden-output and distribution tests before we call them production-safe.

Research and Productionization Notes

1. Speculative Decoding Families

The foundation is target-verified speculative decoding: a small proposer drafts tokens and the full Kimi model verifies them in parallel. The verifier preserves the target distribution when implemented correctly. Relevant systems and papers include SpecInfer, EAGLE, EAGLE-2, EAGLE-3, heterogeneous-vocabulary lossless speculative decoding, and LongSpec for long-context settings.

  • Perun angle: implement a common proposer/verifier interface first, then plug in n-gram, prompt-lookup, small draft model, and EAGLE-style draft heads.
  • AMD issue: verifier tree attention can become expensive if branch packing and KV access are not ROCm-native.
  • Acceptance metric: accepted tokens per target forward pass, not just raw drafter speed.

2. Prefix Caching and Chunk Reuse

Prefix caching skips redundant prefill by reusing exact KV states. This is especially relevant for Kimi K2.6 because long agentic prompts often share system instructions, tool schemas, repository summaries, and retrieved chunks. The 2025 MLSys paper Marconi shows that admission and eviction policy matters, reporting much higher token hit rates and lower TTFT for hybrid models.

  • Perun angle: build request-prefix fingerprints that include model revision, tokenizer, chat template, tool schema, and exact token IDs.
  • AMD issue: 256K context means cache policy must be memory-aware; bad admission can evict high-value decode KV.
  • Production test: replay real agent traces and measure TTFT, HBM pressure, hit rate, and eviction churn.

3. Chunked Prefill and Stall-Free Scheduling

SARATHI and Sarathi-Serve split long prefill into chunks and schedule them alongside decodes to avoid decode stalls. Sarathi-Serve reports throughput gains under latency SLOs compared with Orca and vLLM baselines. Sources: SARATHI and Sarathi-Serve.

  • Perun angle: make chunk sizing adaptive to Kimi prompt length, batch mix, and current decode queue depth.
  • AMD issue: the optimal chunk size may differ from NVIDIA because MI300X has different HBM bandwidth, xGMI topology, and ROCm kernel launch behavior.
  • Production test: measure p50/p95/p99 TTFT and inter-token latency under mixed long-prompt and short-chat traffic.

4. Disaggregated Prefill/Decode and KV Transport

DistServe and Mooncake separate prefill and decode onto different resources, while newer work such as FlowKV focuses on lowering KV transfer overhead. This is exact if the KV cache is transferred without approximation and scheduling semantics are preserved. Sources: DistServe, Mooncake, and FlowKV.

  • Perun angle: prioritize single-node correctness first, then RDMA-backed PD disaggregation for 4-8 GPU AMD nodes.
  • AMD issue: KV transfer and collectives must be tuned for xGMI/RDMA; otherwise decode nodes idle while waiting for cache.
  • Production test: measure goodput under SLO, KV transfer latency, GPU idle time, and memory pressure.

5. AMD-Native MLA and MoE Kernels

AMD's ROCm optimization material describes AITER as a source of ROCm-specific fused kernels for attention, MLA, MoE, GEMM, RMSNorm, RoPE, and KV-cache paths. AMD's Kimi K2.5 blog found fused MoE to be the key bottleneck and used FlyDSL to create a mixed-precision fused MoE kernel for MI300X. Sources: ROCm vLLM optimization guide, ROCm AITER, and AMD Kimi K2.5 fused MoE optimization.

  • Perun angle: build a kernel registry keyed by model family, GPU family, dtype, hidden size, expert count, top-k, and batch tokens.
  • AMD issue: TP=4 is currently the safe Kimi K2.6 MLA layout; TP=8 violates the AITER head-per-GPU constraint documented by SGLang.
  • Production test: compare logits and sampled outputs against upstream kernels across greedy and stochastic decoding seeds.

Do Not Count These as Lossless Yet

Technique Why It Is Risky When to Reconsider
Post-training quantization Usually changes model outputs. Kimi's native INT4 release is acceptable as the baseline, but additional FP8/FP4/W4A8 changes are not automatically lossless. Separate quality/performance track with benchmark gates.
KV eviction, pruning, sparsification, compression Dropping or approximating KV changes attention inputs unless the model architecture explicitly supports it. Only if exact reconstruction or model-native sparse attention semantics are proven.
Early exit Skipping layers changes the target model unless wrapped in a verified speculative scheme. Use only as a proposer inside verified speculative decoding.
Turning off thinking mode It changes the requested generation behavior. It may be a product knob, but it is not lossless for the same request. Expose as a workload-level latency option, not an engine optimization.

Recommended First Experiments

  1. Profile Kimi K2.6 on AMD TP=4: collect per-kernel time for MLA decode, fused MoE, routing/top-k, RMSNorm, RoPE, collectives, sampling, and scheduler overhead.
  2. Port the Kimi K2.5 fused MoE approach: adapt FlyDSL/AITER fused MoE to K2.6 shapes and validate logits against the baseline.
  3. Add prompt-lookup speculative decoding: low-risk proposer, no training needed, strong for code and repeated agent traces.
  4. Build the verified speculation interface: proposer API, verifier API, tree packing, acceptance accounting, and deterministic replay.
  5. Improve prefix-cache policy: add trace-driven admission/eviction that optimizes saved FLOPs per HBM byte.
  6. Tune chunked prefill: sweep chunk sizes against realistic 8K, 32K, 128K, and 256K prompts.
  7. Prototype PD disaggregation only after local kernels are competitive: otherwise the cluster design will hide single-node inefficiencies.
